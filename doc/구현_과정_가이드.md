## 1. 전체 시스템 아키텍처

```
[사용자 브라우저]
    │ https://blog.pluscoach.co.kr
    ▼
[GitHub Pages]
    │ index.html + assets/{css,js}/*
    │ PayApp JS SDK (payapp-lite.js)
    │ Supabase JS CDN (@supabase/supabase-js@2)
    ▼
[Supabase (egwmkpplnzypkbedasrs)]
    │
    ├── create_order RPC (anon 호출)
    │       ↓ INSERT 발생
    ├── orders 테이블
    │       ↓ AFTER INSERT Trigger
    ├── notify_new_order() 트리거 함수
    │       ↓ net.http_post (Authorization: Bearer + Vault service_role_key)
    │
    ▼
[Edge Function: on-new-order]
    │
    ├── plan === 'free_trial' → 3중 알림 + 자동 라이선스 발급 + 인증키 이메일
    │   ├─→ 카카오톡 "나에게 보내기" ─→ 사장님 카톡
    │   ├─→ Telegram Bot API        ─→ 사장님 텔레그램
    │   ├─→ Resend 접수 확인 이메일  ─→ 사용자
    │   ├─→ create_license RPC      ─→ licenses INSERT
    │   └─→ Resend 인증키 이메일    ─→ 사용자 (DOWNLOAD_URL_FREE)
    │
    └── plan !== 'free_trial' (monthly/full_package/lifetime) → 조기 반환 SUCCESS (아무 알림 없음)
        ↓ 프론트가 PayApp 결제창 호출 (payment.js)
        ↓ 사용자 결제 완료
        ↓
[PayApp 서버] → POST payapp-webhook (form-urlencoded)
        ↓
[Edge Function: payapp-webhook]
    ├─→ linkval 검증 + pay_state=4 확인 + 금액 검증
    ├─→ create_license RPC → licenses INSERT
    ├─→ orders UPDATE (license_key + status='결제완료')
    ├─→ Resend 정식판 인증키 이메일 (DOWNLOAD_URL_PAID)
    ├─→ 카카오톡 결제 완료 알림 ─→ 사장님 카톡
    └─→ 텔레그램 결제 완료 알림 ─→ 사장님 텔레그램
```

### 데이터 흐름 — 무료체험

1. 사용자가 폼에서 무료체험 선택 후 제출
2. `form.js`가 `ipify` API로 IP 조회 → `create_order` RPC 호출
3. RPC가 이메일 중복 체크 → `orders` INSERT
4. Trigger → `on-new-order` → 3중 알림(카톡/텔레그램/이메일) + 라이선스 발급 + 인증키 이메일
5. 프론트엔드에서 성공 토스트 표시

### 데이터 흐름 — 유료 결제

1. 사용자가 폼에서 유료 플랜 선택 후 제출
2. `form.js` → `create_order` RPC → `orders` INSERT (status='결제대기')
3. Trigger → `on-new-order` → 유료 조기 반환 (알림 없음)
4. `payment.js` → PayApp 결제창 팝업 (var1=order_code)
5. 사용자 결제 완료 → PayApp 서버가 `payapp-webhook` POST
6. Webhook → linkval 검증 → 금액 검증 → 라이선스 발급 → 인증키 이메일 → 카톡/텔레그램 결제 알림

## 2. 기술 스택과 선택 이유

| 계층 | 기술 | 선택 이유 |
|------|------|----------|
| **프론트엔드** | HTML + Tailwind CDN + Vanilla JS | 빌드 도구 없이 GitHub Pages 즉시 배포. MVP에 프레임워크 불필요 |
| **폰트** | Pretendard (CDN) | 한국어 최적화, Apple SD Gothic Neo 대체 |
| **백엔드** | Supabase (PostgreSQL) | 기존 프로그램 라이선스 시스템과 동일 프로젝트 사용. 무료 티어 |
| **RPC** | plpgsql SECURITY DEFINER | RLS 우회하면서 비즈니스 로직(중복체크, 코드생성) 서버사이드 처리 |
| **Trigger** | pg_net + Vault | DB INSERT 이벤트로 Edge Function 자동 호출. Vault로 키 안전 관리 |
| **Edge Function** | Deno/TypeScript | Supabase 네이티브. 콜드스타트 빠름 |
| **카카오톡 알림** | "나에게 보내기" API | 사장님 카톡에 직접 알림. 비즈니스 메시지 대비 심사 불필요 |
| **텔레그램 알림** | Bot API | 카카오톡 장애 시 백업. 설정 5분 |
| **이메일** | Resend | 개발자 친화적 API, 무료 월 3000건, 도메인 인증 간단 |
| **배포** | GitHub Pages + 카페24 CNAME | 무료. 서브도메인으로 기존 사이트 영향 0 |
| **토큰 관리** | kakao_tokens 테이블 + 자동갱신 | access_token 6시간 만료 → Edge Function에서 자동 refresh |

## 3. Supabase 프로젝트 구조

### 3.1 테이블

| 테이블 | 상태 | RLS | 용도 |
|--------|------|-----|------|
| `licenses` | 기존 (절대 수정 금지) | ON + 정책 있음 | 프로그램 라이선스 관리 |
| `orders` | Phase 3 신규 | ON + 정책 없음 (RPC만 접근) | 주문/신청 관리 |
| `kakao_tokens` | Phase 3 신규 | ON + 정책 없음 (service_role만) | 카카오 토큰 단일 행 |

### 3.2 RPC 함수

| 함수 | 권한 | 설명 |
|------|------|------|
| `verify_and_activate(key, machine_id)` | anon | 기존 — 프로그램 라이선스 검증 |
| `update_session(key, machine_id)` | anon | 기존 — 세션 갱신 |
| `clear_session(key)` | anon | 기존 — 세션 정리 |
| `create_order(p_name, p_email, p_plan, p_amount, p_ip, p_user_agent)` | anon | 신규 — 주문 생성 + 중복체크 |
| `create_license(p_buyer_name, p_plan, p_order_code)` | service_role 전용 | 신규 — 라이선스 키 발급 (Phase 10.5에서 lifetime 추가) |

### 3.3 Trigger

| 트리거 | 테이블 | 이벤트 | 함수 |
|--------|--------|--------|------|
| `on_orders_updated` | orders | BEFORE UPDATE | `handle_updated_at()` — updated_at 자동 갱신 |
| `on_order_inserted` | orders | AFTER INSERT | `notify_new_order()` — Edge Function 호출 |

### 3.4 Extensions

|------|------|
| `pg_net` | Trigger에서 HTTP POST (Edge Function 호출) |

### Kakao 토큰 발급 curl 명령어

```bash
# Step 1: 브라우저에서 인증코드 받기 (REST_API_KEY 교체)
# https://kauth.kakao.com/oauth/authorize?client_id=[REST_API_KEY]&redirect_uri=https://localhost:3000/oauth/callback&response_type=code&scope=talk_message

# Step 2: 인증코드로 토큰 발급
curl -X POST "https://kauth.kakao.com/oauth/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=authorization_code" \
  -d "client_id=[REST_API_KEY]" \
  -d "client_secret=[CLIENT_SECRET]" \
  -d "redirect_uri=https://localhost:3000/oauth/callback" \
  -d "code=[인증코드]"

# Step 3: kakao_tokens 테이블에 INSERT
INSERT INTO public.kakao_tokens (id, access_token, refresh_token, access_token_expires_at)
VALUES (1, '[access_token]', '[refresh_token]', NOW() + INTERVAL '21599 seconds')
ON CONFLICT (id) DO UPDATE SET
  access_token = EXCLUDED.access_token,
  refresh_token = EXCLUDED.refresh_token,
  access_token_expires_at = EXCLUDED.access_token_expires_at,
  updated_at = NOW();
```

### 6.5 Vault current_setting 권한 문제

**증상:** 초기 설계에서 `current_setting('app.settings.service_role_key')` 방식으로 service_role_key를 저장하려 했으나, Supabase 환경에서 `ALTER DATABASE SET` 권한 없음

**원인:** Supabase 매니지드 환경에서 PostgreSQL `current_setting`에 커스텀 변수를 등록하는 `ALTER DATABASE` 명령에 제한이 있음

**해결:** Supabase Vault 사용으로 전환:
```sql
-- 저장
SELECT vault.create_secret('실제_SERVICE_ROLE_KEY', 'service_role_key');

-- 읽기 (Trigger 함수 내에서)
SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = 'service_role_key';
```

**장점:** Vault가 더 안전함 (암호화 저장, 접근 제어). `current_setting`은 평문이었을 것.

### 6.6 RFC 5322 위반 — From 헤더에 한글 직접 삽입 (Phase 6.5)

**증상:** `from: '블로그 자동화 솔루션 <noreply@...>'` — 이메일 수신 시 발신자명이 깨지거나 스팸 판정 가능성

**원인:** 이메일 From 헤더에 비-ASCII 문자(한글)를 직접 넣으면 RFC 5322 위반. Resend API가 받아줘도 수신측 메일 클라이언트에서 문제 발생 가능

**해결:** MIME encoded-word (RFC 2047) 방식으로 인코딩:
```typescript
const FROM_NAME_B64 = btoa(String.fromCharCode(...new TextEncoder().encode("블로그 자동화 솔루션")));
const FROM_HEADER = `=?UTF-8?B?${FROM_NAME_B64}?= <${RESEND_FROM_EMAIL}>`;
```

**검증:** Gmail/네이버메일에서 발신자 "블로그 자동화 솔루션" 정상 표시 확인

### 6.7 create_license RPC 반환값 타입 혼동 (Phase 6)

**증상:** `supabase.rpc('create_license')` 호출 결과를 `data.key`로 접근하려 함

**원인:** create_license RPC가 `RETURNS TEXT` (단일 문자열)이므로, `{ data: "BAF-...", error: ... }` 형태로 반환. `data` 자체가 라이선스 키 문자열

**해결:** `const { data: licenseKey } = ...` 로 직접 사용. `data.key` 같은 객체 접근 금지

**교훈:** RPC의 RETURNS 타입이 TEXT/INTEGER 같은 스칼라인지, JSON/TABLE인지 반드시 확인 후 코딩

### 6.8 status 덮어쓰기 경합 방지 설계 (Phase 6)

**문제:** 무료체험 분기에서 라이선스 발급 → orders UPDATE → 이메일 발송 순서로 진행하는데, 각 단계 실패 시 status를 다르게 설정해야 함. 잘못하면 한 흐름에서 status가 두 번 이상 덮어쓰임

**설계 원칙:**
1. 라이선스 발급 실패 → `status='발급실패'` → **early return** (이후 코드 안 탐)
2. 라이선스 성공 → `status='발송완료'` UPDATE → 이메일 실패 시만 `status='이메일실패'`로 덮어씀
3. DOWNLOAD_URL_FREE 미설정 → `status='이메일실패'` (한 번만)
4. 정상 흐름에서 status가 두 번 이상 덮어쓰이는 경로 없음

**결과:** status 값으로 정확히 어느 단계에서 실패했는지 파악 가능
- `처리중` → 아직 Edge Function이 처리 안 함
- `발급실패` → create_license RPC 에러
- `발송완료` → 정상 완료
- `이메일실패` → 라이선스는 발급됐지만 이메일 못 보냄
- `데이터오류` → 필수 필드 누락

### 6.9 git push rejected — non-fast-forward (Phase 8)

### 6.15 플로팅 버튼 레이아웃 깨짐 — absolute vs flex 충돌 (Phase 11)

**증상:** PC에서 툴팁과 원형 버튼이 분리돼 보이고, 모바일에서 원형 버튼이 아예 안 보임

**원인:** `.kakao-float`이 `display: flex` 컨테이너인데, 자식 `.kakao-float-tooltip`을 `position: absolute`로 배치. flex 컨테이너 안에서 absolute 자식은 flex flow에서 제외되어 위치 계산이 의도대로 안 됨. 또한 `position: fixed`가 Tailwind CSS에 의해 덮어쓰일 가능성.

**해결:**
1. `position: fixed !important` — 우선순위 확보
2. 툴팁을 `position: relative`로 변경 + flex `gap: 12px`로 원형과 자연스럽게 가로 배치
3. SVG에 `display: block` 명시

**교훈:** flex 컨테이너 안에서 자식을 `position: absolute`로 배치하면 flex flow와 충돌. flex `gap`으로 간격을 잡는 것이 더 안정적.

### 8.3 Database Trigger → Edge Function 패턴

```sql
-- 1. Extensions
CREATE EXTENSION IF NOT EXISTS pg_net WITH SCHEMA extensions;
CREATE EXTENSION IF NOT EXISTS supabase_vault WITH SCHEMA vault;

-- 2. Vault에 키 저장
SELECT vault.create_secret('[SERVICE_ROLE_KEY]', 'service_role_key');

-- 3. Trigger 함수
CREATE OR REPLACE FUNCTION public.notify_on_insert()
RETURNS TRIGGER AS $$
DECLARE v_key TEXT;
BEGIN
    SELECT decrypted_secret INTO v_key
    FROM vault.decrypted_secrets WHERE name = 'service_role_key' LIMIT 1;

    PERFORM net.http_post(
        url := 'https://[PROJECT_REF].supabase.co/functions/v1/[FUNCTION_NAME]',
        headers := jsonb_build_object(
            'Content-Type', 'application/json',
            'Authorization', 'Bearer ' || v_key
        ),
        body := jsonb_build_object('record', row_to_json(NEW)::jsonb)
    );
    RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- 4. Trigger 연결
CREATE TRIGGER on_table_inserted
    AFTER INSERT ON public.[TABLE_NAME]
    FOR EACH ROW EXECUTE FUNCTION public.notify_on_insert();
```
