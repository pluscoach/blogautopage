## 3. 핵심 아키텍처 결정사항

### 3.1 결제 모드 스위치 (뺐다 꼈다 원칙)

**프론트**: `assets/js/config.js`
```javascript
window.APP_CONFIG = {
    paymentMode: 'bank_transfer',  // 'bank_transfer' | 'payapp'
    bankInfo: {
        bank: '하나은행',
        account: '120-910375-30807',
        holder: '오준석(제이에스코퍼레이션)'
    },
    // ... 기존 supabase, kakao, site 등
};
```

**백엔드**: Supabase Edge Function Secret
```bash
supabase secrets set PAYMENT_MODE=bank_transfer  # 또는 payapp
```

**페이앱 복귀 시나리오**:
```
1. config.js 수정: paymentMode: 'payapp'
2. supabase secrets set PAYMENT_MODE=payapp
3. git push
→ 2~3분 내 완전 복귀. 무통장 코드는 dormant 상태로 보존.
```

### 3.2 두 단계 알림 구조 + 답장 연결

**왜 이렇게 설계됐나**:
- 고객의 "폼 제출"과 "입금 완료 신고"를 **명확히 구분**
- 여러 주문이 동시에 들어와도 **답장 UI**로 시각적 구분 가능
- 첫 번째 알림에 이미 [승인] 버튼이 있어 **고객 빨간 버튼 대기 불필요** — 입금만 확인되면 즉시 처리

**두 알림의 차이**:

| 알림 | 발송 시점 | 발송자 | 버튼 | 제목 |
|------|-----------|--------|------|------|
| 첫 번째 | 폼 제출 시 | `on-new-order` | ✅ 있음 | `💳 신규 주문 (입금 확인 필요)` |
| 두 번째 | 빨간 버튼 클릭 시 | `notify-deposit-confirmed` | ❌ 없음 | `✅ 고객이 입금 완료 버튼을 눌렀습니다.` (답장 형태) |

### 3.3 멱등성 보장

**중복 승인 방지**:
- `handleApprove`: `order.license_key` 이미 채워져 있으면 "이미 발급됨" 토스트 + 스킵
- 같은 주문에서 [승인] 여러 번 눌러도 안전

**중복 입금 신고 알림 방지**:
- `notify-deposit-confirmed`: `order.status === '입금대기'` 이미 이면 중복 발송 스킵
- 고객이 빨간 버튼 여러 번 눌러도 사장님에겐 알림 1회만

### 3.4 재발송 기능 설계

**원칙**: **기존 라이선스 재사용**
- 이메일만 다시 발송, 라이선스 키는 동일
- `licenses` 테이블에 row 추가 생성 없음
- 기존 이메일이 스팸 처리됐거나 실패한 경우 용도

**다른 이메일로 재발송 시**:
- `orders.email`은 **업데이트 안 함** (원본 신청 기록 유지)
- 단순히 새 이메일 주소로 발송만

**Telegram force_reply 방식**:
- 봇이 "📧 주문 {orderCode} 새 이메일 주소를 회신해주세요" 메시지 발송
- Telegram이 자동으로 회신 UI 띄움
- 사장님이 회신으로 새 이메일 입력 → 봇이 `reply_to_message` 컨텍스트로 주문 식별
- Stateless (세션 DB 불필요)

**시그니처**: 6파라미터 → 7파라미터 (맨 뒤에 `p_phone` optional 추가)
```sql
create_order(
    p_name TEXT,
    p_email TEXT,
    p_plan TEXT,
    p_amount INTEGER,
    p_ip TEXT DEFAULT NULL,
    p_user_agent TEXT DEFAULT NULL,
    p_phone TEXT DEFAULT ''

```

## 13. 빠른 디버깅 가이드

### 13.1 주문 생성됐는데 텔레그램 알림 안 옴
```sql
-- 1. 주문이 DB에 들어갔는지 확인
SELECT id, name, status, telegram_notice_message_id, created_at
FROM orders ORDER BY id DESC LIMIT 5;

-- 2. telegram_notice_message_id NULL이면 → on-new-order 실패
-- Supabase Dashboard → Edge Functions → on-new-order → Logs 확인
```

**가능한 원인**:
- `PAYMENT_MODE` Secret 미설정 → `'payapp'` fallback → 알림 안 감
- Telegram Bot Token 무효화
- `sendDepositNoticeWithButtons` 내부 에러

### 13.2 고객이 빨간 버튼 눌렀는데 두 번째 알림 안 옴
```sql
-- 1. orders.status 확인
SELECT id, name, status, telegram_notice_message_id, order_code
FROM orders WHERE order_code = '주문코드-XXXX';

-- 2. 기대 값:
-- - status = '입금대기' (UPDATE 완료)
-- - telegram_notice_message_id 값 있음
```

**가능한 원인**:
- `telegram_notice_message_id` NULL → 첫 번째 알림 발송 실패 → 답장 연결 불가
- `notify-deposit-confirmed` Logs 확인

### 13.3 승인 버튼 눌렀는데 이메일 안 옴
```sql
-- 1. 라이선스 발급 여부
SELECT id, name, email, status, license_key FROM orders WHERE order_code = '...';

-- 2. license_key 채워졌으면 → 발급 OK, 이메일 발송 단계 실패
-- status 확인:
-- - '결제완료': 정상
-- - '이메일실패': 이메일만 실패 (수동 재발송 가능)
-- - '발급실패': create_license 실패 (긴급 이메일 받았을 것)
```
