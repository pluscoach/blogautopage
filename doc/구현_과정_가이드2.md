## 2. 데이터 흐름

### 무료체험
1. 사용자 폼 제출 → `create_order` RPC → INSERT (status='처리중')
2. Trigger → on-new-order → 3중 알림 + 라이선스 발급 + 인증키 이메일
3. 프론트 성공 토스트

### 유료 무통장 입금 (현재 활성)
1. 사용자 폼 제출 (이름/이메일/전화/플랜) → `create_order` RPC → INSERT (status='결제대기')
2. Trigger → on-new-order → 사장님 텔레그램 알림 (버튼 포함) + message_id 저장
3. 프론트 무통장 모달 표시 (계좌 정보 + 빨간 버튼)
4. 고객 빨간 버튼 → notify-deposit-confirmed → status='입금대기' + 답장 알림
5. 사장님 텔레그램 [✅ 승인] → telegram-callback → 라이선스 발급 + 이메일 발송

### 유료 페이앱 결제 (PAYMENT_MODE='payapp' 시)
1. 사용자 폼 제출 → `create_order` → INSERT (status='결제대기')
2. Trigger → on-new-order → 조기 반환
3. 프론트 PayApp 결제창 → 결제 완료 → payapp-webhook → 라이선스 + 이메일

### Step 1: DB Migration — orders.phone + create_order 7파라미터

**파일:** `supabase/migrations/20260418120000_add_phone_and_extend_create_order.sql`
**커밋:** `5cb80c4`

**작업 내용:**
- `ALTER TABLE orders ADD COLUMN phone TEXT DEFAULT ''` — 기존 row는 빈 문자열 자동 채움
- `CREATE OR REPLACE FUNCTION create_order(...)` — 7파라미터 (p_phone 맨 뒤, DEFAULT '')
- 기존 6파라미터 함수 `DROP` + 7파라미터에 `GRANT EXECUTE TO anon, authenticated`
- 비즈니스 로직(중복 체크, order_code 생성, INSERT)은 기존 원본과 100% 동일

**설계 결정:**
- p_phone을 맨 뒤 optional로 추가하여 기존 호출부(form.js의 named parameter) 호환 유지
- PostgreSQL named parameter notation이라 기존 6파라미터 호출도 에러 없이 작동
- GRANT 시그니처가 변경되므로(7파라미터) 기존 6파라미터 GRANT는 무효 → 명시적 DROP 필수

**검증:** 6파라미터 호출(기존 호환), 7파라미터 호출(phone 포함), 중복 체크 에러, phone 저장 확인 — 9개 검증 쿼리 전부 통과
